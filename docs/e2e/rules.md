# Playwright 化ルール

## 仕様管理

- `test.md` はリリース前の棚卸しチェックリストとして使う。
- Playwright に落とす項目は `docs/e2e/release-test-spec.md` に `E2E-XXX-999` の一意IDで詳細化する。
- 各ケースには「前提データ」「手順」「期待値」「対応テストファイル」「手動確認の残項目」を書く。
- Playwright のテスト名には必ずケースIDを含める。
- テストコードには対応仕様へのコメントを置く。

## セレクタ

優先順位は以下の通りです。

1. `getByRole`
2. `getByLabel`
3. `getByText`
4. `data-testid`
5. CSS セレクタ

CSS セレクタは、意味のある role/name/label で取れない場合だけ使います。動的な DB ID、Bootstrap の見た目クラス、DOM 階層の深い指定には依存しません。

## テストデータ

- テストは独立して実行できる単位にする。
- 破壊的操作はテスト専用ユーザーとテスト専用データに限定する。
- 認証情報は環境変数 `E2E_USER_EMAIL` と `E2E_USER_PASSWORD` で渡す。
- 認証情報が必要なテストは原則失敗させる。公開ページだけ確認したい場合に限り `E2E_ALLOW_AUTH_SKIP=1` でスキップを許可する。
- 既存データに依存する期待値は避け、画面表示、遷移、作成したデータの有無を確認する。
- 作成・編集などユーザー導線の確認が目的のケースは、フォームUIから入力して送信する。APIそのものの検証が目的のケースだけ `page.request` を使う。
- テストが作成したデータは、同じテスト内で削除する。削除できない履歴系データはテスト専用ユーザーをリセットしてから実行する。

## 期待値

- URL 遷移、見出し、主要 CTA、フォームエラー、成功メッセージ、一覧への反映を明示的に検証する。
- `console.error` と `pageerror` は原則失敗扱いにする。
- レスポンシブ確認は 390px 幅を初期自動化対象にし、360px、768px、実機相当の細かい見た目は手動確認に残す。
- 色、余白、完全な視覚崩れなど、人の判断が必要な項目は md に手動確認として残す。

## スコープ

- 初期自動化はスモークテストを優先する。
- CRUD は主要機能の作成、一覧反映、状態切替、削除までを自動化対象にする。細かい編集UI、長押し、ドラッグなど入力デバイス依存の操作は手動確認に残す。
- Safari/iOS Safari 相当、Android 実機、実メール送信、cron 実行結果は初期 Playwright では対象外にし、別手順で確認する。
